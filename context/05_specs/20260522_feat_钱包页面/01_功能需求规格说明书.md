# 功能需求规格说明书

## 1. 概述

### 1.1 功能背景

Ocean API 平台作为 AI API 网关，向用户提供按量计费的 API 调用服务。用户需要一个集中的资金管理页面来查看余额、充值额度、使用兑换码以及查看充值历史。钱包页面是用户与平台计费系统的核心交互入口。

### 1.2 本功能业务目标

1. 提供清晰的账户余额概览，让用户实时了解资金状态
2. 支持多种支付渠道的在线充值，覆盖国内外主流支付方式
3. 提供兑换码充值能力，支持离线/批量充值场景
4. 提供完整的充值历史查询，保障交易透明度

## 3. 页面与功能总览

### 3.1 页面清单

| 序号 | 页面名称 | 页面形式 | 职责说明 | 包含功能 |
|-----|---------|---------|---------|---------|
| 1 | 钱包页面 | 独立页面 | 用户资金管理的主入口，展示余额概览和充值操作 | 余额查看、在线充值、兑换码充值、账单历史入口 |
| 2 | 支付确认弹窗 | 弹窗对话框 | 确认在线充值的金额、折扣和支付方式后发起支付 | 充值金额确认、折扣展示、支付发起 |
| 3 | Creem产品确认弹窗 | 弹窗对话框 | 确认 Creem 加密货币产品的购买详情后跳转结账 | 产品信息确认、结账跳转 |
| 4 | 账单历史弹窗 | 弹窗对话框 | 展示充值交易记录列表，支持搜索和分页 | 交易记录查询、管理员补单 |

### 3.2 页面跳转流程

```
[入口：侧边栏/导航 - Wallet] → [钱包页面]
                                      ├─ 点击"支付方式"按钮 → [支付确认弹窗] → 确认 → 跳转第三方支付页面
                                      ├─ 点击"Creem产品" → [Creem产品确认弹窗] → 确认 → 跳转 Creem 结账页面
                                      ├─ 点击"Order History"按钮 → [账单历史弹窗]
                                      └─ 兑换码输入 + 点击"Redeem" → 直接完成（不打开弹窗）
```

**流程说明：**
1. 用户通过导航菜单进入钱包页面
2. 页面加载时自动获取用户余额数据和充值配置
3. 用户选择充值金额和支付方式后，系统弹出支付确认弹窗
4. 用户确认后跳转到第三方支付页面完成支付
5. 支付完成后，第三方通过 Webhook 异步回调系统，系统更新订单状态并增加用户余额
6. 用户从第三方支付页面返回时，系统通过 `show_history` URL 参数自动打开账单历史弹窗（如 `/wallet?show_history=true`），并刷新用户余额数据
7. 用户可随时打开账单历史弹窗查看交易记录

## 4. 页面功能详细说明

### 4.1 页面 1：钱包页面

#### 4.1.1 页面概述

**页面形式：**独立页面

**页面职责：**用户资金管理的核心页面，上方展示账户余额概览（当前余额、总使用量、API请求总数），下方提供充值操作表单（在线充值和兑换码充值）。用户日常充值操作的主要入口。

#### 4.1.2 涉及字段

##### A. 钱包概览区域字段

| 字段名称 | 字段类型 | 数据来源 | 获取时机 | 业务含义 |
|---------|---------|---------|---------|--------|
| 当前余额（quota） | 标签 | 数据库查询 | 页面加载 | 用户当前可用的 API 调用额度，单位为配额（500,000 配额 = $1） |
| 总使用量（used_quota） | 标签 | 数据库查询 | 页面加载 | 用户累计已消耗的 API 调用额度 |
| API请求总数（request_count） | 标签 | 数据库查询 | 页面加载 | 用户累计发起的 API 请求次数 |

##### B. 充值表单区域字段

| 字段名称 | 字段类型 | 数据来源 | 获取时机 | 校验规则 | 默认值 | 业务含义 |
|---------|---------|---------|---------|---------|--------|--------|
| 预设金额选项 | 选择按钮组 | 系统预设（由后端 amount_options 配置） | 页面加载 | 无 | 第一个预设金额 | 快捷选择充值金额，点击即选中并计算实付金额 |
| 自定义金额 | 数字输入框 | 用户输入 | 实时输入 | >= 当前支付渠道最小充值金额；整数 | 当前支付渠道最小充值金额 | 用户自定义充值金额 |
| 支付方式 | 按钮组 | 系统预设（由后端 pay_methods 配置） | 页面加载 | 无 | 第一个可用支付方式 | 选择充值所使用的支付渠道 |
| Waffo支付方式 | 按钮组 | 系统预设（由后端 waffo_pay_methods 配置） | 页面加载 | 无 | 无 | Waffo 渠道的子支付方式选择 |
| Creem产品 | 卡片列表 | 系统预设（由后端 creem_products 配置） | 页面加载 | 无 | 无 | Creem 加密货币渠道的固定面额产品列表 |
| 兑换码 | 文本输入框 | 用户输入 | 实时输入 | 非空 | 空 | 输入兑换码进行充值 |
| 实付金额 | 标签（计算字段） | 支付渠道金额计算接口返回（POST /api/user/amount、/stripe/amount、/waffo-pancake/amount） | 充值金额或支付方式变化时 | 无 | 无 | 前端调用后端金额计算接口获取实付金额，非本地计算 |

**充值配置开关字段（由后端 TopupInfo 返回，控制页面区域可见性）：**

| 配置字段 | 类型 | 业务含义 |
|---------|------|---------|
| enable_online_topup | 布尔 | 是否启用易支付在线充值 |
| enable_stripe_topup | 布尔 | 是否启用 Stripe 信用卡充值 |
| enable_creem_topup | 布尔 | 是否启用 Creem 加密货币充值 |
| enable_waffo_topup | 布尔 | 是否启用 Waffo 充值 |
| enable_waffo_pancake_topup | 布尔 | 是否启用 Waffo Pancake 充值 |
| enable_redemption | 布尔 | 是否启用兑换码功能 |
| payment_compliance_confirmed | 布尔 | 是否已完成支付合规确认 |

#### 4.1.3 功能与按钮

**用户可操作功能：**

- **选择预设金额**
  - 触发按钮：预设金额卡片按钮
  - 按钮级别：字段级
  - 按钮位置：充值表单 "Amount" 区域
  - 触发方式：点击按钮
  - 权限要求：见 2.2 页面访问权限
  - 加载状态：无
  - 功能说明：点击预设金额卡片后，选中该金额并自动计算实付金额。预设金额卡片上展示充值金额、实付金额和折扣标签（如有折扣）

- **输入自定义金额**
  - 触发方式：在自定义金额输入框中输入数值
  - 按钮级别：字段级
  - 权限要求：见 2.2 页面访问权限
  - 加载状态：无

- **选择标准支付方式并发起支付**

  - 触发按钮：Waffo 支付方式按钮
  - 按钮级别：字段级

  - 按钮级别：字段级
  - 按钮位置：充值表单区域
  - 触发方式：点击按钮
  - 权限要求：见 2.2 页面访问权限
  - 加载状态：点击后显示 loading

- **选择 Creem 产品并发起支付**
  - 触发按钮：Creem 产品卡片
  - 按钮级别：字段级
  - 按钮位置：充值表单 "Creem Payment" 区域

  - 权限要求：见 2.2 页面访问权限
  - 加载状态：无（在确认弹窗中处理 loading）

  - 触发方式：点击按钮

**后台自动流程：**

- **页面初始化加载**
  - 触发时机：页面加载时
  - 说明：自动获取用户信息（余额、使用量、请求次数）和充值配置信息（支付方式、预设金额、折扣率、功能开关）

- **兑换码获取链接展示**
  - 触发时机：页面加载完成，topup_link 配置存在时
  - 说明：在兑换码区域下方显示"Need a redemption code? Get one here"外部链接

- **支付返回后余额刷新**
  - 触发时机：用户从第三方支付页面返回钱包页面时

#### 4.1.4 业务规则

- **规则1：折扣系统**
  - 规则描述：系统支持按充值金额设置不同的折扣率，通过 discount 配置映射（key=充值金额，value=折扣率）。折扣率范围 0~1，1 表示无折扣。折扣信息在预设金额卡片和支付确认弹窗中展示
  - 触发条件：充值金额在 discount 映射中存在对应配置
  - 约束说明：折扣按金额整数值匹配（key=int(amount)），预设金额和自定义金额均可享受折扣；实际业务中通常仅对预设金额配置折扣

- **规则2：最小充值金额校验**
  - 规则描述：每种支付渠道有独立的最小充值金额限制（min_topup、stripe_min_topup、waffo_min_topup、waffo_pancake_min_topup）。当前充值金额低于某个支付方式的最小金额时，该支付方式按钮置灰
  - 触发条件：用户修改充值金额时
  - 约束说明：自定义金额输入框的 placeholder 显示当前支付渠道的最小充值金额

- **规则2.5：订单过期时间**
  - 规则描述：各支付渠道的订单过期时间不同，无统一配置项。Stripe 由平台控制（通常 24 小时），超时触发 checkout.expired 事件回调将订单标记为 expired；Waffo Pancake 硬编码 45 分钟超时；易支付和 Creem 无服务端超时机制，订单将保持 pending 状态直到被手动处理
  - 触发条件：订单创建后计时

- **规则3：支付渠道可用性控制**
  - 规则描述：页面的各支付区域是否可见取决于后端返回的功能开关（enable_online_topup、enable_stripe_topup 等）。当所有在线充值渠道都关闭时，显示提示"Online topup is not enabled"
  - 触发条件：页面加载获取充值配置后
  - 约束说明：兑换码功能可通过 enable_redemption 独立控制，与在线充值互不影响

- **规则4：支付方式选择与金额计算联动**
  - 规则描述：用户选择预设金额或修改自定义金额后，系统自动调用对应支付渠道的金额计算接口获取实付金额。不同支付渠道（易支付、Stripe、Waffo Pancake）有各自的金额计算接口
  - 触发条件：充值金额变化时
  - 约束说明：金额计算过程中显示 loading 骨架屏

- **规则5：兑换码验证**
  - 规则描述：兑换码提交后，后端验证兑换码的状态（必须为未使用）和过期时间（必须未过期）。验证通过后直接增加用户余额，使用数据库行锁保证并发安全
  - 触发条件：用户点击"Redeem"按钮
  - 约束说明：兑换码为一次性使用，成功后状态变为已使用

- **规则6：配额换算**
  - 规则描述：系统内部配额单位与美元的换算率为 500,000 配额 = $1。前端展示时根据系统配置的 quotaDisplayType 和 usdExchangeRate 进行换算显示
  - 触发条件：所有涉及金额/配额展示的场景
  - 约束说明：当 quotaDisplayType 为 USD 时，直接以美元展示

- **规则7：支付合规控制**
  - 规则描述：当 payment_compliance_confirmed 为 false 时，兑换码区域显示提示"Redemption codes are disabled until the administrator confirms compliance terms"，兑换码输入框和按钮不可用
  - 触发条件：页面加载获取充值配置后
  - 约束说明：此开关由超级管理员在系统设置中控制

#### 4.1.5 交互逻辑

- 预设金额卡片采用 2 列（移动端）/ 4 列（桌面端）网格布局，选中状态的卡片有边框和背景色变化
- 自定义金额输入框旁实时显示"Amount to pay:"和计算中的骨架屏或实付金额
- Creem 产品区域以独立卡片列表形式展示，与标准支付方式区域通过分隔线区分
- 兑换码区域通过分隔线与在线充值区域区分
- 当在线充值全部关闭时，整个在线充值区域替换为 Alert 提示
- 页面加载期间，概览卡片和充值表单均显示骨架屏

### 4.2 页面 2：支付确认弹窗

#### 4.2.1 页面概述

**页面形式：**弹窗对话框（AlertDialog）

**页面职责：**用户选择支付方式后弹出，展示充值的详细确认信息（充值金额、实付金额、折扣信息、支付方式），用户确认后发起实际支付。

#### 4.2.2 涉及字段

| 字段名称 | 字段类型 | 数据来源 | 获取时机 | 校验规则 | 默认值 | 业务含义 |
|---------|---------|---------|---------|---------|--------|--------|
| 充值金额（Topup Amount） | 标签 | 页面状态传递 | 弹窗打开 | 无 | 无 | 用户选择的充值金额，以本地货币显示 |
| 实付金额（You Pay） | 标签 | 金额计算接口返回 | 弹窗打开 | 无 | 无 | 实际需要支付的金额，有折扣时显示折扣后金额 |

## 5. 非页面功能详细说明

### 5.1 支付回调 Webhook 处理

#### 5.1.1 功能概述

**触发方式：**API调用（第三方支付平台回调）
**触发时机：**用户在第三方支付平台完成支付后，平台异步回调系统
**功能职责：**验证支付回调签名，确认支付成功后更新订单状态并增加用户配额

#### 5.1.2 处理流程

1. 接收第三方支付平台的 Webhook 请求
2. 验证回调签名和请求合法性
3. 根据 trade_no 查找对应的 pending 状态订单
4. 更新订单状态为 success，记录完成时间
5. 增加用户配额（IncreaseUserQuota）
6. 返回成功响应给支付平台

#### 5.1.3 涉及数据

| 数据项 | 数据来源 | 数据用途 | 说明 |
|-------|---------|---------|------|
| trade_no | Webhook 请求参数 | 定位订单 | 支付平台的订单号 |
| 签名信息 | Webhook 请求头/参数 | 验证合法性 | 各支付平台有独立的签名验证机制 |
| 支付状态 | Webhook 请求参数 | 判断支付结果 | 不同平台的状态字段名称不同 |
| 订单记录 | 数据库查询 | 业务处理 | TopUp 表中对应 trade_no 的记录 |

#### 5.1.4 业务规则

- **规则1：**签名验证失败的回调直接拒绝，记录异常日志
- **规则2：**已成功订单的重复回调直接返回成功，不重复加款（幂等处理）
- **规则3：**各支付渠道（易支付/Stripe/Creem/Waffo）有独立的 Webhook 路由和签名验证逻辑

#### 5.1.5 异常处理

| 异常场景 | 处理方式 | 错误提示/日志 |
|---------|---------|--------------|
| 签名验证失败 | 拒绝请求，返回错误 | 记录签名不匹配日志 |
| 订单不存在 | 拒绝请求 | 记录无效 trade_no |
| 订单已成功 | 返回成功（幂等） | 无 |
| 订单已过期（expired） | 拒绝请求，不执行加款 | 记录过期订单收到回调日志 |
| 订单已失败（failed） | 拒绝请求，不执行加款 | 记录失败订单收到回调日志 |
| 数据库写入失败 | 返回失败，等待重试 | 记录错误日志 |

#### 5.1.6 与其他功能的关联

- **上游依赖：**第三方支付平台（易支付/Stripe/Creem/Waffo）发起回调
- **下游影响：**用户配额增加，影响钱包余额展示

## 6. 数据状态定义

### 6.1 数据状态

| 状态名称 | 状态说明 | 可转换到的状态 |
|---------|---------|--------------|
| pending | 待支付：用户已发起充值，等待在第三方支付平台完成支付 | success、expired、failed |
| success | 支付成功：用户完成支付，配额已到账 | 无（终态） |
| failed | 支付失败：第三方确认支付未成功（如 Waffo 非成功终态） | 无（终态） |
| expired | 已过期：充值订单超时未支付 | 无（终态） |

### 6.2 状态转换规则

| 当前状态 | 可转换到的状态 | 转换条件 | 转换触发方式 |
|---------|------------|---------|-------------|
| pending | success | 第三方支付平台回调确认支付成功，或管理员手动补单 | 系统自动（Webhook回调）/ 用户操作（管理员补单） |
| pending | expired | 订单超过有效期未支付（Stripe: 由平台控制约24小时；Waffo Pancake: 45分钟） | 系统自动（Stripe expired 事件回调）/ 系统自动（Waffo Pancake 超时） |
| pending | failed | 第三方支付平台回调确认支付失败（如 Waffo 非 PAY_SUCCESS 终态） | 系统自动（Webhook回调） |
