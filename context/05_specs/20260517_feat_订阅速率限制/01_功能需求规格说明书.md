# 功能需求规格说明书

## 1. 概述

### 1.1 功能背景

系统已实现订阅套餐功能，用户购买套餐后获得固定周期内的额度使用权。当前计费系统仅控制总额度消耗，不限制 token 消耗的速率。这意味着用户可以在短时间内集中消耗大量 token，导致上游 API 成本骤增，影响服务稳定性。

主流 AI coding plan 产品（Cursor、Claude Pro、OpenAI）均采用速率限制来控制资源消耗节奏。本功能在现有订阅系统基础上，为 coding plan 类型套餐增加基于 token 的速率限制能力。

### 1.2 本功能业务目标

1. 为 coding plan 套餐提供 token 级别的速率限制，控制用户在每个时间窗口内的 token 消耗速率
2. 采用 5 小时固定窗口 + 自然周两个维度限制，5 小时窗口控制突发用量，周窗口控制持续用量
3. 在 Relay 请求链路中预消费之前执行限速检查，超限时返回 429 错误并携带详细限速信息
4. 通过 API 响应头和钱包页面向用户展示限速状态，提升透明度
5. 在套餐浏览和购买环节展示限速参数，帮助用户在购买前了解使用限制

### 3.2 页面跳转流程

```
[管理员侧]
管理后台 → 管理员订阅管理页 → 点击"创建/编辑套餐" → 套餐编辑抽屉（含套餐类型选择与动态字段）

[用户侧]
钱包页 → 订阅区域 → 浏览套餐卡片（API 套餐与 CodingPlan 套餐分区域展示）→ 点击"立即订阅" → 购买确认弹窗（按类型展示对应参数）→ 选择支付方式 → 跳转支付
                → 查看限速状态（CodingPlan 订阅展示当前窗口已用/剩余 token、下次重置时间）
```

**流程说明：**
1. 管理员在套餐编辑抽屉中选择套餐类型，系统根据类型动态展示对应配置字段。API 套餐展示总额度、额度重置等字段；CodingPlan 套餐展示 5 小时 token 上限、周倍数字段
2. 用户在钱包页浏览套餐时，API 套餐卡片与 CodingPlan 套餐卡片分区域展示，各自显示对应类型的参数信息
3. 用户点击购买时，购买确认弹窗按套餐类型展示对应参数：API 套餐展示 token 额度，CodingPlan 套餐展示限速参数
4. 用户购买 CodingPlan 后在钱包页订阅区域查看限速使用状态

## 4. 页面功能详细说明

### 4.1 页面 1：套餐编辑抽屉（套餐类型与动态字段）

#### 4.1.1 页面概述

**页面形式：**现有抽屉内调整字段布局

**页面职责：**在现有套餐创建/编辑抽屉中，调整字段布局。套餐类型置于套餐副标题下方，单独占一行；实付金额、升级分组、限购、排序、启用状态、有效期为所有套餐类型的固定字段；根据套餐类型动态展示对应配置字段：API 套餐展示总额度与额度重置相关字段，CodingPlan 套餐展示速率限制参数。

#### 4.1.2 涉及字段

##### A. 固定字段（所有套餐类型均展示）

| 字段名称 | 字段类型 | 数据来源 | 获取时机 | 校验规则 | 默认值 | 业务含义 |
|---------|---------|---------|---------|---------|--------|---------|
| 套餐类型 | 单选 | 用户选择 | 页面加载 | 必填 | API 套餐 | 套餐类型：API 套餐 或 CodingPlan 套餐。置于套餐副标题下方，单独占一行 |
| 实付金额 | 数字 | 用户输入 | 实时输入 | 必填，≥ 0 | 无 | 套餐实际售价 |
| 升级分组 | 文本 | 用户输入 | 实时输入 | 可选 | 无 | 套餐所属升级分组 |
| 限购 | 数字 | 用户输入 | 实时输入 | 可选，≥ 0 | 无 | 每位用户限购次数 |
| 排序 | 数字 | 用户输入 | 实时输入 | 可选，整数 | 无 | 套餐展示顺序 |
| 启用状态 | 开关 | 用户选择 | 页面加载 | 必填 | 启用 | 套餐是否上架 |
| 有效期 | 数字 | 用户输入 | 实时输入 | 必填，≥ 1，整数 | 无 | 套餐有效天数 |

##### B. API 套餐动态字段（仅套餐类型为 API 套餐时展示）

| 字段名称 | 字段类型 | 数据来源 | 获取时机 | 校验规则 | 默认值 | 业务含义 |
|---------|---------|---------|---------|---------|--------|---------|
| 总额度 | 数字 | 用户输入 | 实时输入 | API 套餐时必填，≥ 0.01，允许小数 | 无 | 套餐总可用 token 额度。输入以百万（M）为基础单位，默认单位 MTokens |
| 额度重置 | 组合 | 用户选择/输入 | 实时输入 | API 套餐时必填 | 无 | 额度重置周期及相关配置 |

##### C. CodingPlan 套餐动态字段（仅套餐类型为 CodingPlan 套餐时展示）

| 字段名称 | 字段类型 | 数据来源 | 获取时机 | 校验规则 | 默认值 | 业务含义 |
|---------|---------|---------|---------|---------|--------|---------|
| 5 小时 Token 上限 | 数字 | 用户输入 | 实时输入 | CodingPlan 套餐时必填，≥ 0.01，允许小数 | 无 | 任意固定 5 小时窗口内的 token 消耗上限。输入以百万（M）为基础单位，默认单位 MTokens |
| 周倍数 | 数字 | 用户输入 | 实时输入 | CodingPlan 套餐时必填，≥ 1，整数 | 无 | 周上限 = 5 小时上限 × 周倍数，控制一周内的总消耗量 |

**字段说明：**
- 套餐类型为 API 套餐时，仅展示 B 类动态字段（总额度、额度重置），隐藏 C 类字段
- 套餐类型为 CodingPlan 套餐时，仅展示 C 类动态字段（5 小时 Token 上限、周倍数），隐藏 B 类字段
- 总额度与 5 小时 Token 上限的输入框均以百万（M）为基础单位，前端展示默认单位 MTokens。例如管理员输入 500，实际表示 500,000,000 tokens（500M）
- 周上限由系统自动计算（5 小时上限 × 周倍数），不在表单中单独配置，实时展示计算结果
- 示例：5 小时 Token 上限输入 0.5（即 500,000 tokens，0.5M），周倍数 = 20，则周上限 = 10,000,000 tokens（10M）

#### 4.1.3 功能与按钮

**用户可操作功能：**

- **配置套餐参数**
  - 触发按钮：复用现有"保存"/"更新"按钮
  - 按钮级别：页面级（现有）
  - 按钮位置：抽屉底部（现有）
  - 触发方式：点击按钮
  - 权限要求：管理员
  - 加载状态：是
  - 功能说明：固定字段与动态字段（API 套餐的额度配置或 CodingPlan 的限速参数）随套餐表单一起提交保存，无需独立保存按钮

#### 4.1.4 业务规则

- **规则1：动态字段按套餐类型必填**
  - 规则描述：套餐类型为 API 套餐时，总额度为必填；套餐类型为 CodingPlan 时，5 小时 Token 上限和周倍数为必填。不允许部分配置
  - 触发条件：保存套餐时
  - 约束说明：前端表单校验 + 后端校验双重保障

- **规则2：周上限自动计算**
  - 规则描述：周上限 = 5 小时上限 × 周倍数，表单中实时展示计算结果
  - 触发条件：任一字段变更时
  - 约束说明：前端实时计算展示，后端保存时计算并存储周上限值

- **规则3：动态字段显示条件**
  - 规则描述：套餐类型为 API 套餐时，展示总额度、额度重置相关字段，隐藏 5 小时 Token 上限和周倍数；套餐类型为 CodingPlan 时，展示 5 小时 Token 上限和周倍数，隐藏总额度、额度重置相关字段
  - 触发条件：套餐类型变更或页面加载时
  - 约束说明：前端根据套餐类型动态控制字段区域的显示与隐藏

- **规则4：套餐配置变更不影响已有订阅**
  - 规则描述：管理员修改套餐的限速参数或额度后，仅影响新购买的订阅。已有订阅保留购买时的配置快照
  - 触发条件：套餐配置被修改时
  - 约束说明：订阅创建时将套餐类型、限速参数等快照到 UserSubscription 中，后续限速检查使用订阅上的快照值而非套餐上的最新值

#### 4.1.5 交互逻辑

- 套餐类型字段置于套餐副标题下方，单独占一行，与下方表单区域视觉分隔
- 套餐类型变更时，动态切换下方动态字段区域：API 套餐展示总额度与额度重置相关字段，CodingPlan 套餐展示 5 小时 Token 上限与周倍数
- 总额度与 5 小时 Token 上限的输入框右侧默认展示单位"MTokens"，输入值以百万为基础单位（输入 1 表示 1,000,000 tokens）
- 5 小时 Token 上限和周倍数字段变更时，实时展示计算后的周上限预览值（如"周上限：10,000,000 tokens"）
- 动态字段区域位于固定字段之后，与固定字段视觉分隔

### 4.2 页面 2：用户钱包页（限速状态展示）

#### 4.2.1 页面概述

**页面形式：**现有内嵌组件内新增展示区域

**页面职责：**在用户钱包页的订阅区域中，展示当前活跃订阅的限速使用情况，包括 5 小时窗口和周窗口的 token 消耗进度。

#### 4.2.2 涉及字段

##### A. 显示字段

| 字段名称 | 字段类型 | 数据来源 | 获取时机 | 业务含义 |
|---------|---------|---------|---------|---------|
| 5 小时窗口已用 token | 标签 | Redis 计数器查询 | 页面加载 | 当前 5 小时窗口内已消耗的 token 数 |
| 5 小时窗口剩余 token | 标签 | 计算值 | 页面加载 | 5 小时上限 - 已用 token |
| 5 小时窗口重置时间 | 标签 | 计算 | 页面加载 | 当前 5 小时窗口结束的时间点 |
| 周窗口已用 token | 标签 | Redis 计数器查询 | 页面加载 | 当前自然周内已消耗的 token 数 |
| 周窗口剩余 token | 标签 | 计算值 | 页面加载 | 周上限 - 已用 token |
| 周窗口重置时间 | 标签 | 计算 | 页面加载 | 当前自然周结束的时间点（下周一 00:00） |
| 限速进度条 | 进度条 | 计算值 | 页面加载 | 已用/上限的百分比，5 小时和周各一个 |

#### 4.2.3 功能与按钮

**用户可操作功能：**

- **查看限速状态**
  - 触发方式：页面自动加载
  - 权限要求：所有登录用户
  - 功能说明：自动查询当前活跃订阅的限速使用情况并展示

#### 4.2.4 业务规则

- **规则1：多订阅独立展示**
  - 规则描述：用户有多个活跃订阅时，分别展示每个订阅的限速状态
  - 触发条件：页面加载时
  - 约束说明：每个订阅的限速独立计算，互不影响

- **规则2：非 CodingPlan 套餐不展示限速区域**
  - 规则描述：套餐类型为 API 套餐的订阅不展示限速状态区域
  - 触发条件：页面加载时
  - 约束说明：仅 CodingPlan 套餐的订阅才展示限速进度

- **规则3：无活跃订阅时不展示**
  - 规则描述：用户无活跃订阅时，限速状态区域整体不展示
  - 触发条件：页面加载时
  - 约束说明：复用现有「无活跃订阅」提示

#### 4.2.5 交互逻辑

- token 数量展示时自动格式化：超过 1000 时显示为 "X.XK"，超过 1,000,000 时显示为 "X.XM"
- 重置时间显示为相对时间（如"2 小时后重置"）和绝对时间（如"2026-05-17 15:00"）

### 4.3 页面 3：套餐卡片（按类型展示参数）

#### 4.3.1 页面概述

**页面形式：**现有套餐卡片内调整展示

**页面职责：**在钱包页的套餐浏览区域中，API 套餐与 CodingPlan 套餐分区域展示。根据套餐类型在卡片上展示对应参数：API 套餐展示 token 额度，CodingPlan 套餐展示限速参数，帮助用户在购买前了解使用限制。

#### 4.3.2 涉及字段

##### A. 显示字段

| 字段名称 | 字段类型 | 数据来源 | 获取时机 | 业务含义 |
|---------|---------|---------|---------|---------|
| Token 额度 | 标签 | 数据库（TotalAmount） | 页面加载 | API 套餐的总可用 token 额度 |
| 每 5 小时可用额度 | 标签 | 数据库（RateLimitTokensPerWindow） | 页面加载 | CodingPlan 套餐的 5 小时窗口 token 消耗上限 |
| 每周可用额度 | 标签 | 计算值（RateLimitTokensPerWindow × RateLimitWeeklyMultiplier） | 页面加载 | CodingPlan 套餐的自然周 token 消耗上限 |

#### 4.3.3 功能与按钮

**用户可操作功能：**

- **查看套餐参数**
  - 触发方式：页面自动加载
  - 权限要求：所有登录用户
  - 功能说明：套餐卡片按类型自动展示对应参数，无需额外操作

#### 4.3.4 业务规则

- **规则1：API 套餐与 CodingPlan 套餐分区域展示**
  - 规则描述：钱包页套餐浏览区域将 API 套餐与 CodingPlan 套餐分为两个独立区域展示，各自按顺序排列
  - 触发条件：页面加载时
  - 约束说明：前端根据套餐类型（PlanType）分组渲染，每类套餐独立成区

- **规则2：CodingPlan 套餐不展示总额度**
  - 规则描述：套餐类型为 CodingPlan 的卡片不展示总额度信息，仅展示限速参数（每 5 小时可用额度、每周可用额度）
  - 触发条件：页面加载时
  - 约束说明：CodingPlan 套餐的额度概念由限速参数替代，卡片上不显示传统总额度字段

- **规则3：API 套餐展示 Token 额度**
  - 规则描述：套餐类型为 API 套餐的卡片展示 Token 额度（即总额度），展示格式与 CodingPlan 的"每 5 小时可用额度"一致
  - 触发条件：页面加载时
  - 约束说明：API 套餐卡片展示总额度值，不展示限速参数

- **规则4：参数展示位置**
  - 规则描述：参数展示在卡片现有信息（价格、有效期、升级组）之后、"立即订阅"按钮之前
  - 触发条件：卡片渲染时
  - 约束说明：参数不替换卡片上的已有信息，作为补充展示

#### 4.3.5 交互逻辑

- API 套餐卡片以带说明文字的格式展示 Token 额度，格式为"Token 额度：X tokens"
- CodingPlan 套餐卡片以带说明文字的格式展示限速参数，格式为"每 5 小时可用额度：X tokens"和"每周可用额度：X tokens"
- token 数量展示时自动格式化：超过 1000 时显示为"X.XK"，超过 1,000,000 时显示为"X.XM"（如"Token 额度：10M tokens"、"每 5 小时可用额度：500K tokens"、"每周可用额度：10M tokens"）

### 4.4 页面 4：购买确认弹窗（按类型展示参数）

#### 4.4.1 页面概述

## 5. 非页面功能详细说明

### 5.1 限速检查

#### 5.1.1 功能概述

**触发方式：**API 请求时自动触发
**触发时机：**用户通过 AI API 发起请求时，在预消费（PreConsume）之前执行
**功能职责：**检查用户当前活跃订阅的 token 消耗是否超过限速窗口的上限，超限时拒绝请求。

#### 5.1.2 处理流程

1. 用户发起 API 请求，进入 Relay 处理链路
2. 识别用户是否有配置了限速的活跃订阅
3. 如非 CodingPlan 订阅，跳过检查，继续后续计费流程
4. 如有限速订阅，查询 Redis 中该订阅的当前 5 小时窗口和周窗口的 token 已用量
5. 判断是否超限：
   a. 5 小时窗口已用 ≥ 5 小时上限 → 拒绝，返回 429
   b. 周窗口已用 ≥ 周上限 → 拒绝，返回 429
   c. 未超限 → 继续后续预消费流程
6. 在 API 响应头中附加限速状态信息（X-RateLimit-*）

#### 5.1.3 涉及数据

| 数据项 | 数据来源 | 数据用途 | 说明 |
|-------|---------|---------|------|
| UserSubscription.PlanType（快照值） | 数据库 | 套餐类型（API 套餐 / CodingPlan 套餐） | 决定限速配置是否生效，取订阅创建时的快照值 |
| UserSubscription.RateLimitTokensPerWindow（业务名：5 小时 token 上限，快照值） | 数据库 | 5 小时窗口 token 上限 | 取订阅创建时快照到 UserSubscription 的值 |
| UserSubscription.RateLimitWeeklyMultiplier（业务名：周倍数，快照值） | 数据库 | 周倍数 | 取订阅创建时快照到 UserSubscription 的值，计算周上限 = 上限 × 倍数 |
| Redis Sorted Set: rate_limit:{subscription_id}:5h:{window_key} | Redis | 5 小时窗口内每次请求的 token 消耗记录 | score = timestamp, member = request_id:token_count |
| Redis Sorted Set: rate_limit:{subscription_id}:week:{week_key} | Redis | 周窗口内每次请求的 token 消耗记录 | 同上 |

#### 5.1.4 业务规则

- **规则1：限速检查在预消费之前执行**
  - 限速检查不扣费，仅判断是否允许请求。通过后才进入预消费流程
  - 超限时直接返回 429，不创建预消费记录

- **规则2：5 小时窗口划分**
  - 以 UTC 0:00 为起点，每 5 小时为一个固定窗口
  - 窗口边界：[0:00, 5:00), [5:00, 10:00), [10:00, 15:00), [15:00, 20:00), [20:00, 24:00)
  - 最后一个窗口为 4 小时，但 token 上限不变（与 5 小时窗口相同的额度）
  - 固定窗口的已知 trade-off：窗口边界处（如 4:59 和 5:01）允许在短时间内消耗接近 2 倍配额。这是可接受的，因为现有 RPM 中间件已在分钟级别控制了请求突发频率

- **规则3：周窗口划分**
  - 以自然周为固定窗口，UTC 周一 00:00 到 UTC 周一 00:00
  - 周上限 = 5 小时上限 × 周倍数

- **规则4：双窗口独立判断**
  - 5 小时窗口和周窗口独立判断，任一窗口超限即拒绝
  - 两个窗口的判断不存在先后顺序

- **规则5：非 CodingPlan 套餐跳过检查**
  - 订阅关联的套餐类型非 CodingPlan 时，跳过限速检查
  - 非订阅计费模式（钱包支付）的用户不受限速影响

- **规则6：与现有 ModelRequestRateLimit 中间件的分工**
  - 现有 `middleware/model-rate-limit.go` 已实现每分钟 RPM（请求数）限速，使用 Token Bucket 算法，在 relay-router.go:73 执行（认证之后、分发之前）
  - 本功能的 TPM（token 消耗量）限速是独立的第二层控制，不替代现有 RPM 限速
  - 两者关系：RPM 限速控制请求频率（防止短时间大量请求），TPM 限速控制 token 消耗速率（防止资源过度消耗）
  - 执行顺序：RPM 限速（现有）→ TPM 限速（本功能）→ 预消费 → 上游请求
  - 两个限速独立判断，任一超限即拒绝请求

#### 5.1.5 异常处理

| 异常场景 | 处理方式 | 错误提示/日志 |
|---------|---------|--------------|
| Redis 不可用 | 降级放行请求，记录告警日志 | 告警：rate limit check skipped due to Redis unavailable |
| 限速配置数据异常（上限为 0 或负数） | 降级放行请求，记录错误日志 | 错误：invalid rate limit config for plan {id} |

#### 5.1.6 与其他功能的关联

- **上游依赖：**UserSubscription 的限速快照值、Redis 计数器数据
- **下游影响：**限速通过后继续 Relay 预消费流程；限速拒绝时直接返回 429，不触发预消费

### 5.2 限速用量记录

#### 5.2.1 功能概述

**触发方式：**API 请求完成后自动触发
**触发时机：**流式响应结束或非流式响应返回后
**功能职责：**将请求的实际 token 消耗量记录到 Redis Sorted Set 中，作为限速检查的数据来源。

#### 5.2.2 处理流程

1. API 请求完成（流式结束或非流式响应），获取实际 token 消耗量
2. 计算当前 5 小时窗口 key（基于 UTC 时间的窗口编号）和周窗口 key（基于自然周的周编号）
3. 向 Redis Sorted Set 中添加记录：
   a. `rate_limit:{subscription_id}:5h:{window_key}`: score = timestamp, member = request_id:token_count
   b. `rate_limit:{subscription_id}:week:{week_key}`: score = timestamp, member = request_id:token_count
4. 设置 TTL：5 小时窗口 key 的 TTL = 6 小时（留余量），周窗口 key 的 TTL = 8 天

#### 5.2.3 涉及数据

| 数据项 | 数据来源 | 数据用途 | 说明 |
|-------|---------|---------|------|
| 实际 token 消耗量 | Relay 响应处理 | 记录到限速计数器 | 输入 token + 输出 token |
| Redis Sorted Set | Redis | 存储限速计数 | 每次请求添加一条记录 |

#### 5.2.4 业务规则

- **规则1：仅记录 CodingPlan 套餐的订阅**
  - 请求计费走订阅且订阅套餐类型为 CodingPlan 时，才记录限速用量
  - 走钱包计费或套餐类型为 API 套餐的订阅，不记录

- **规则2：流式请求事后统计**
  - 流式请求在全部输出完成后，根据实际 token 消耗量一次性记录
  - 不做预估或预扣，限速以事后统计为准

- **规则3：记录失败不影响请求结果**
  - Redis 写入失败时，记录错误日志，不影响已完成的 API 请求
  - 限速精度可能暂时降低，但保证服务可用性

- **规则4：并发超额的容忍策略**
  - 本功能采用事后统计模型，接受并发流式请求导致的窗口内超额
  - 超额不触发惩罚（不回溯拒绝已完成请求），但下次限速检查时会基于实际累计用量判断
  - 现有 RPM 中间件（ModelRequestRateLimit）控制并发请求数量，从请求频率层面间接限制并发超额幅度
  - 这是有意的 trade-off：选择实现简洁性和流式体验完整性，接受限速精度约为 ±1 个并发请求的 token 消耗量

#### 5.2.5 异常处理

| 异常场景 | 处理方式 | 错误提示/日志 |
|---------|---------|--------------|
| Redis 写入失败 | 记录错误日志，不影响请求 | 错误：failed to record rate limit usage for subscription {id} |

#### 5.2.6 与其他功能的关联

- **上游依赖：**Relay 响应处理（获取实际 token 消耗量）
- **下游影响：**记录的用量数据供限速检查（5.1）查询使用

### 5.3 限速响应头注入

#### 5.3.1 功能概述

**触发方式：**API 请求响应时自动触发
**触发时机：**每次 API 响应返回时
**功能职责：**在 API 响应头中注入限速状态信息，供客户端感知限速状态。

#### 5.3.2 处理流程

1. 用户有限速配置的活跃订阅时，在响应头中注入以下字段：
   - `X-RateLimit-Limit-5h`: 5 小时窗口 token 上限
   - `X-RateLimit-Remaining-5h`: 5 小时窗口剩余 token
   - `X-RateLimit-Reset-5h`: 5 小时窗口重置时间（Unix 时间戳）
   - `X-RateLimit-Limit-Week`: 周窗口 token 上限
   - `X-RateLimit-Remaining-Week`: 周窗口剩余 token
   - `X-RateLimit-Reset-Week`: 周窗口重置时间（Unix 时间戳）
2. 用户非 CodingPlan 套餐时，不注入限速响应头

#### 5.3.3 业务规则

- **规则1：响应头为只读信息**
  - 响应头仅用于信息展示，不影响限速判断逻辑
  - Remaining 值不反映当前进行中的请求消耗量，仅反映已完成请求的累计消耗。实际消耗量可能因并发请求而超出 Remaining 值

- **规则2：429 响应体格式**
  - 超限返回 429 时，响应体包含：
    ```json
    {
      "error": {
        "type": "rate_limit_exceeded",
        "message": "Rate limit exceeded for the current window",
        "rate_limit": {
          "limit_type": "5h",
          "limit": 500000,
          "remaining": 0,
          "reset_at": 1747461600
        }
      }
    }
    ```
  - 同时附加 `Retry-After` 响应头，值为当前窗口剩余秒数

#### 5.3.4 异常处理

| 异常场景 | 处理方式 | 错误提示/日志 |
|---------|---------|--------------|
| Redis 不可用 | 跳过响应头注入，不影响正常请求响应 | 告警：rate limit header injection skipped due to Redis unavailable |
| 用户非 CodingPlan 套餐 | 不注入限速响应头，正常返回请求响应 | 无 |

### 5.4 限速数据清理

#### 5.4.1 功能概述

**触发方式：**事件触发
**触发时机：**订阅过期或取消时
**功能职责：**清理该订阅在 Redis 中的限速计数器数据。

#### 5.4.2 处理流程

1. 订阅状态变更为 expired 或 cancelled 时触发
2. 查询该订阅在 Redis 中所有限速相关的 Sorted Set key
3. 批量删除这些 key
4. 记录清理日志

#### 5.4.3 涉及数据

| 数据项 | 数据来源 | 数据用途 | 说明 |
|-------|---------|---------|------|
| Redis Sorted Set keys | Redis | 待清理的限速计数器 | 按 subscription_id 匹配 |

#### 5.4.4 业务规则

- **规则1：按订阅粒度清理**
  - 仅清理过期/取消订阅的限速数据，不影响同一用户其他活跃订阅
  - 清理操作通过 Redis key pattern 匹配实现

- **规则2：清理失败不阻塞过期流程**
  - Redis 清理失败时记录错误日志，不阻塞订阅过期/取消的主流程
  - 依赖 key 的 TTL 机制兜底（6 小时 / 8 天后自动过期）

#### 5.4.5 异常处理

| 异常场景 | 处理方式 | 错误提示/日志 |
|---------|---------|--------------|
| Redis 不可用 | 记录错误日志，依赖 TTL 兜底 | 错误：failed to cleanup rate limit data for subscription {id} |
| Key 匹配异常 | 记录错误日志，继续后续流程 | 错误：rate limit key pattern scan failed for subscription {id} |

#### 5.4.6 与其他功能的关联

- **上游依赖：**订阅过期处理（定时任务）、管理员作废操作
- **下游影响：**清理后该订阅的限速数据不再占用 Redis 内存

### 5.5 限速状态查询 API

#### 5.5.1 功能概述

**触发方式：**API 调用
**触发时机：**钱包页加载或客户端主动查询时
**功能职责：**返回当前用户活跃订阅的限速使用情况，供前端展示和客户端工具使用。

#### 5.5.2 处理流程

1. 接收 GET /api/subscription/rate-limits 请求
2. 查询用户所有活跃订阅
3. 对每个有限速配置的订阅，查询 Redis 中的限速用量
4. 计算当前窗口的剩余 token 和重置时间
5. 返回限速状态数据

#### 5.5.3 涉及数据

| 数据项 | 数据来源 | 数据用途 | 说明 |
|-------|---------|---------|------|
| UserSubscription | 数据库 | 活跃订阅列表 | 筛选 active 状态 |
| UserSubscription（限速快照字段） | 数据库 | 限速配置参数 | 取订阅创建时快照的 5 小时上限、周倍数 |
| Redis Sorted Set | Redis | 实际 token 消耗 | 聚合窗口内用量 |

#### 5.5.4 业务规则

- **规则1：仅返回 CodingPlan 套餐的订阅**
  - 套餐类型为 API 套餐的订阅不出现在返回结果中
  - 避免返回无意义的限速数据

- **规则2：窗口用量实时计算**
  - 通过 ZRANGEBYSCORE 聚合当前窗口内的所有记录
  - 计算剩余 token = 上限 - 已用

#### 5.5.5 异常处理

| 异常场景 | 处理方式 | 错误提示/日志 |
|---------|---------|--------------|
| Redis 不可用 | 返回空数组，记录告警日志 | 告警：rate limit status query skipped due to Redis unavailable |
| 用户无活跃订阅 | 返回空数组 | 无 |
| 查询 Redis 失败 | 返回空数组，记录错误日志 | 错误：failed to query rate limit usage for subscription {id} |

## 6. 数据状态定义

本功能不引入新的数据状态。限速计数器为无状态的数值记录，不存在状态流转。

限速功能与订阅系统共享以下已有状态：

| 关联数据 | 状态 | 限速行为 |
|---------|------|---------|
| UserSubscription | active | 执行限速检查和用量记录 |
| UserSubscription | expired | 停止限速检查，触发数据清理 |
| UserSubscription | cancelled | 停止限速检查，触发数据清理 |

## 7. 集成和依赖

### 7.1 外部系统集成

| 系统名称 | 集成方式 | 集成内容 | 调用时机 | 依赖功能 |

| 订阅速率限制 | Redis 基础设施 | 限速计数器使用 Redis Sorted Set 存储 | 数据依赖 |
| 订阅速率限制 | 套餐浏览接口（GET /api/subscription/plans） | 套餐卡片和购买弹窗展示限速参数，需要该接口返回 RateLimitTokensPerWindow 和 RateLimitWeeklyMultiplier 字段 | 数据依赖 |

| 5 小时窗口 | 以 UTC 0:00 为起点的固定 5 小时时间段 | 窗口边界：0-5, 5-10, 10-15, 15-20, 20-24 |
| 周窗口 | 以 UTC 周一 00:00 为起点的自然周 | 与日历周对齐 |
| 周倍数 | 周上限相对于 5 小时上限的倍数 | 周上限 = 5 小时上限 × 周倍数 |
| Token 限速 | 限制用户在特定时间窗口内的 token 消耗量 | 输入 token + 输出 token 合计 |
| Redis Sorted Set | Redis 有序集合数据结构 | 用于存储限速计数器，支持按时间范围聚合查询 |
