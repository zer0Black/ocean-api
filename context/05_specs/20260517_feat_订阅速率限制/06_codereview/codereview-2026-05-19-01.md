## 问题

### 严重（必须修复）

（无）

### 重要（应该修复）

1. **RecordRateLimitUsage 每次 ZADD 都重置 TTL**
   - 文件: `service/rate_limit.go:171,177`
   - 问题: Pipeline 中每次 ZADD 后都执行 `Expire`，这意味着即使 key 已存在且 TTL 很长，也会被重置为固定值（6h / 8天）。如果用户在窗口即将过期时发起请求，TTL 会被刷新回完整长度，导致过期数据延迟清理。
   - 为什么重要: 虽然当前 5h 窗口的 TTL（6h）留有 1h 余量，窗口过期后 key 最多延迟 1h 被清理，这在正常情况下可接受。但周窗口的 TTL 设为 8 天，如果在周尾发起请求，TTL 被重置为 8 天意味着数据可能保留到下一周结束之后，导致 `queryWindowUsage` 读到已属于上一周的数据（但因为 weekKey 不同，实际上读的是新一周的 key，所以不会误读）。
   - 评估: 经过仔细分析，由于 5h 窗口和周窗口的 key 中都包含了窗口标识（windowKey / weekKey），不同窗口的数据天然隔离。TTL 重置只会影响同一窗口内的数据保留时间，不会导致跨窗口数据污染。实际影响是已过期窗口的 key 在 Redis 中多保留一些时间，占用少量内存。这不是功能正确性问题，但属于资源浪费。
   - 建议: 可以在 ZADD 之前先检查 key 是否存在 TTL，仅在 key 新创建时设置 Expire。或接受当前行为，通过自然过期 + TTL 余量控制内存占用。此问题优先级降低为改进项。

2. **queryWindowUsage 的 ZRangeByScore 使用 `-inf` 到 `+inf` 范围**
   - 文件: `service/rate_limit.go:129-132`
   - 问题: 当前查询使用 `-inf` 到 `+inf` 获取 sorted set 的所有成员。由于 TTL 机制保证过期窗口的 key 会被清理，理论上不会有过期数据残留。但如果 TTL 重置（如上条所述）导致旧数据残留，`-inf` 到 `+inf` 会把所有数据都读出来。
   - 为什么重要: 设计文档中指定了 `ZRANGEBYSCORE key <window_start> <window_end>` 的操作方式，使用精确的窗口时间范围更严谨。但由于 TTL 和 key 中的窗口标识已经构成了双重保护，实际上 `-inf` 到 `+inf` 与精确范围的查询结果在这个场景下是等价的。
   - 建议: 可以改为使用精确的窗口时间范围（如 5h 窗口用 `windowStart` 到 `windowEnd`），但这会增加代码复杂度。当前实现在功能上是正确的，优先级降低为改进项。

3. **RecordRateLimitUsage 中 Expire 被无条件调用导致 Pipeline 每次产生 4 个命令**
   - 文件: `service/rate_limit.go:165-178`
   - 问题: 每次 `RecordRateLimitUsage` 调用都发送 2 个 ZADD + 2 个 Expire。在窗口生命周期内，同一个 key 的 Expire 被重复设置多次。虽然 Pipeline 保证了原子性提交，但浪费了 Redis 的 CPU。
   - 建议: 可以使用 `ZADD` + 维护一个本地缓存记录哪些 key 已设置过 TTL，首次写入时才设 Expire。或者接受当前实现，因为 4 个 Pipeline 命令的开销很小。

### 次要（改进项）

1. **测试中直接操作 miniredis 的 ZAdd API 与 go-redis 客户端的 ZAdd 存在差异**
   - 文件: `service/rate_limit_test.go:250-251,288,312,406-407,441-442`
   - 问题: 测试使用 `mr.ZAdd(key, score, member)` 直接操作 miniredis，而实际代码使用 go-redis 客户端的 `ZAdd(ctx, key, &redis.Z{...})`。miniredis 的 ZAdd 参数签名为 `(key, score, member)`，member 格式是 `req_1:100` 这种带冒号的字符串。这是正确的测试方式，但需注意 miniredis 的 ZAdd 行为与 go-redis 在边缘情况下可能存在差异（如 member 已存在时更新 score）。
   - 影响: 低。miniredis 是广泛使用的 Redis mock 库，行为与真实 Redis 高度一致。

2. **RateLimitStatus 结构体中 PlanTitle 和 PlanType 字段在 CheckSubscriptionRateLimit 中未填充**
   - 文件: `service/rate_limit.go:66-72,78-80`
   - 问题: `CheckSubscriptionRateLimit` 返回的 `RateLimitStatus` 中 `PlanTitle` 和 `PlanType` 始终为空字符串。这两个字段需要调用方（如 T4 的 `GetUserRateLimitStatus`）额外填充。当前设计是有意为之（函数职责单一，不查数据库），但调用方容易遗漏。
   - 建议: 在注释中明确说明这两个字段需要调用方填充，或在 `CheckSubscriptionRateLimit` 中添加参数。当前注释已经说明函数只返回状态和是否超限，可以接受。

3. **requestId 中包含冒号的潜在风险**
   - 文件: `service/rate_limit.go:162`
   - 问题: member 格式为 `{requestId}:{tokenCount}`，使用 `strings.SplitN(m, ":", 2)` 解析（`rate_limit.go:141`）。如果 requestId 本身包含冒号，`SplitN(..., 2)` 会正确分割（只按第一个冒号分割），所以这不是 bug。但 requestId 的格式约束没有在函数文档中说明。
   - 建议: 在 `RecordRateLimitUsage` 的函数注释中说明 requestId 不应包含冒号（或说明冒号是安全的因为使用了 SplitN）。

4. **CleanupRateLimitData 中 SCAN 的 COUNT 参数 100 可能需要调优**
   - 文件: `service/rate_limit.go:197`
   - 问题: 每个订阅的 rate limit key 数量有限（最多 5 个 5h 窗口 key + 1 个周 key = 6 个），SCAN COUNT 100 是合理的。但如果清理一个长期活跃的订阅，可能只有 6 个 key，一次 SCAN 就能完成。
   - 影响: 极低。COUNT 100 在这个场景下绰绰有余。
