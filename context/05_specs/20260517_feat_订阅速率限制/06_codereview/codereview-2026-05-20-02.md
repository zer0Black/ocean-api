### 问题

#### 重要

1. **Controller 错误处理不符合项目惯例**
   - 文件：controller/subscription.go:415-417
   - 问题：当 `service.GetUserRateLimitStatus` 返回 error 时，Controller 返回的是 `common.ApiSuccess(c, []interface{}{})`（HTTP 200 + success=true + 空数组），而不是返回错误信息。
   - 为什么重要：项目中所有其他 Controller 函数在遇到数据库/内部错误时，使用 `common.ApiError(c, err)` 返回失败响应（HTTP 200 + success=false + error message）。当前实现对客户端隐藏了内部错误，客户端无法区分"用户确实没有 CodingPlan 订阅"和"服务端出错了"。这违反了 CLAUDE.md 规则 8 的精神（虽然这里是错误响应格式而非硬编码文案）。
   - 如何修复：将 `common.ApiSuccess(c, []interface{}{})` 改为 `common.ApiError(c, err)`。空的限速状态列表和内部错误应该有不同的响应语义。

2. **Service 层 N+1 查询：每个订阅单独查询套餐信息**
   - 文件：service/rate_limit.go:234
   - 问题：`GetUserRateLimitStatus` 在循环内对每个订阅调用 `model.GetSubscriptionPlanById(sub.PlanId)`，这是一个带缓存的查询。虽然缓存减轻了数据库压力，但如果缓存未命中（冷启动场景），N 个订阅会产生 N 次数据库查询。
   - 为什么重要：大多数用户只会有 1-2 个 CodingPlan 订阅，实际影响有限。但在缓存冷启动或缓存失效的边界场景下，这是一个值得注意的查询模式。
   - 如何修复：当前使用缓存的方案在大多数场景下可接受。如果未来订阅数量增长，可以预加载所有 planId 批量查询。暂不阻塞合并，作为后续优化项。

3. **Controller 返回类型 `[]interface{}{}` 与 `[]service.RateLimitStatus{}` 不一致**
   - 文件：controller/subscription.go:416
   - 问题：错误路径返回 `[]interface{}{}`（空 interface 数组），正常路径返回 `[]service.RateLimitStatus{}`。两者的 JSON 序列化结果虽然相同（都是 `[]`），但类型不一致降低了代码可读性。
   - 为什么重要：如果未来错误路径需要返回非空数据（虽然不太可能），类型不匹配可能导致序列化差异。
   - 如何修复：如果保留错误时返回空数组的逻辑，改为统一类型 `[]service.RateLimitStatus{}`。但更推荐按问题 1 的方案修复为 `common.ApiError`。

   - 影响：轻微代码整洁性问题，不阻塞合并。

2. **`setupTestDB` 设置全局 `model.DB` 和 `common` 标志，测试隔离依赖执行顺序**
   - 文件：service/rate_limit_test.go:620-646
   - 问题：`setupTestDB` 直接修改 `model.DB`、`common.UsingSQLite`、`common.RedisEnabled` 等全局变量。虽然 `t.Cleanup` 会关闭数据库连接，但不会恢复原始的 `model.DB` 和 `common` 标志值。这意味着运行顺序可能影响其他测试（实际上已经观察到 service 包全量运行时有一些不相关的测试失败，虽然 T4 测试本身全部通过）。
   - 为什么重要：这是测试框架层面的已有问题，不是 T4 引入的。但 T4 的 `setupTestDB` 加剧了全局状态污染的风险。
   - 如何修复：在 `t.Cleanup` 中恢复 `model.DB`、`common.UsingSQLite`、`common.RedisEnabled` 的原始值。参考 `setupMiniredis` 函数的模式（保存原始值，cleanup 时恢复）。
