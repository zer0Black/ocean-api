# 功能需求规格说明书

## 1. 概述

### 1.1 功能背景

> [修改] 原文：系统已实现订阅套餐功能，用户购买套餐后获得固定周期内的额度使用权。当前计费系统仅控制总额度消耗，不限制 token 消耗的速率。这意味着用户可以在短时间内集中消耗大量 token，导致上游 API 成本骤增，影响服务稳定性。
>
> **变更为**：系统已实现订阅套餐和速率限制功能。现有套餐编辑表单未按套餐类型区分字段布局，所有字段平铺展示，管理员配置不同类型套餐时需要关注不相关的字段。钱包页面的套餐展示也未区分 API 套餐和 CodingPlan 套餐的差异化信息。本次优化重构表单布局（按类型分组字段、统一 token 单位），并优化钱包页套餐展示（分区域展示、按类型差异化显示）。

### 1.2 本功能业务目标

> [修改] 替换基线业务目标为本次变更目标

1. 重构套餐编辑表单，按套餐类型分组显示字段（固定字段 + 类型专属字段），减少管理员配置时的认知负担
2. 统一 token 相关字段（总额度、5小时Token上限）的输入和展示单位为 M token（百万token），提升输入效率和展示可读性
3. 优化钱包页面套餐展示，按 API 套餐和 CodingPlan 套餐分区域展示，不同类型展示差异化信息

## 8. 附录

### 8.1 术语表

> [修改] 新增M token相关术语

| 术语 | 定义 | 说明 |
|------|------|------|
| **[新增]** M token | 百万token的缩写，作为token数量的标准展示和输入单位 | 1 M token = 1,000,000 tokens。用于总额度和5小时Token上限的输入和展示，提升可读性。规范：单位名称用"M token"，展示值用"M tokens"（如"500M tokens"）。M token格式是 K/M 格式化规则在百万级别的具体应用，实现时统一使用一套 token 数量格式化函数 |
| CodingPlan | 套餐类型为 CodingPlan 的订阅套餐 | 通过 SubscriptionPlan.PlanType 字段标识，值为 CodingPlan |
| 5 小时窗口 | 以 UTC 0:00 为起点的固定 5 小时时间段 | 窗口边界：0-5, 5-10, 10-15, 15-20, 20-24 |
| 周窗口 | 以 UTC 周一 00:00 为起点的自然周 | 与日历周对齐 |
| 周倍数 | 周上限相对于 5 小时上限的倍数 | 周上限 = 5 小时上限 × 周倍数 |
| Token 限速 | 限制用户在特定时间窗口内的 token 消耗量 | 输入 token + 输出 token 合计 |
| Redis Sorted Set | Redis 有序集合数据结构 | 用于存储限速计数器，支持按时间范围聚合查询 |
