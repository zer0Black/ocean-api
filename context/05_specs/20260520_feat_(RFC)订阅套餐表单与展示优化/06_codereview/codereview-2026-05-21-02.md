### 问题

#### 严重（必须修复）

无。

#### 重要（应该修复）

1. **API Plan 和 Coding Plan 卡片模板存在大量重复代码**

   - 文件：subscription-plans-card.tsx:595-677（API Plan 卡片）和 714-796（Coding Plan 卡片）
   - 问题：两个分组的 Card 模板（标题、副标题、价格、benefits 渲染、购买按钮、限购提示、tooltip）约 80% 代码完全相同。除了 benefits 数组构建逻辑不同，Card 的 header、price display、StatusBadge、Button/Tooltip 部分逐行一致。
   - 为什么重要：后续修改 Card 布局或样式时需要同时修改两处，容易遗漏导致两种类型的卡片视觉不一致。代码量也从 108 行增长到 232 行（+115%），其中大部分是重复。
   - 修复建议：将 Card 渲染逻辑提取为内部组件（如 `PlanCard`），通过 props 传入 benefits 和 plan 数据。各分组的 IIFE 只负责过滤 plans 和构建不同的 benefits 数组。这样 Card 结构只需维护一处。

2. **IIFE 模式降低了代码可读性**

   - 文件：subscription-plans-card.tsx:562-677, 680-796
   - 问题：使用 `{(() => { ... })()}` 的 IIFE 模式在 JSX 中嵌套了 filter + 条件渲染 + map 的多层逻辑，每个 IIFE 约 115 行。两个 IIFE 各自包含独立的 filter、early return、map 和完整 Card 渲染，嵌套深度达到 6-7 层。
   - 为什么重要：阅读和调试时需要在大量嵌套的大括号间跳转。git diff 也因为两块并行代码而难以对比。如果将 IIFE 提取为独立函数，代码结构会更扁平。
   - 修复建议：将每个 IIFE 提取为组件内的 render 函数，如 `renderApiPlansSection(plans, planPurchaseCountMap, t)` 和 `renderCodingPlansSection(plans, planPurchaseCountMap, t)`，在 JSX 中直接调用。

#### 次要（改进项）

1. **变量命名风格不一致**
   - 文件：subscription-plans-card.tsx:574 vs 697-698
   - 问题：API Plan 分组中定义了 `const totalAmount = Number(plan.total_amount || 0)` 作为中间变量；Coding Plan 分组中 `tokensPerWindow` 和 `weeklyMultiplier` 也做了类似提取。但两者的命名风格和是否提取中间变量的策略不完全一致。API Plan 不提取 `tokensPerWindow` 等变量（因为不需要），Coding Plan 不提取 `totalAmount`（因为不使用）。这实际上是根据各自需要合理组织的，只是两个分组的代码结构缺乏对称性。
   - 影响：纯风格问题，不影响功能。

2. **`formatTokenCount(tokensPerWindow * weeklyMultiplier)` 无溢出防护**
   - 文件：subscription-plans-card.tsx:706
   - 问题：两个 Number 相乘在 JS 中最大安全整数为 2^53-1。正常场景下 token 数不会接近此上限，但如果后端返回异常大的值，乘积可能丢失精度。
   - 影响：实际风险极低，仅作为防御性编程的提醒。
