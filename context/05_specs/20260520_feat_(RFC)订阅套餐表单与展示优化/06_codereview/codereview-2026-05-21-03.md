### 问题

#### 严重（必须修复）

无。

#### 重要（应该修复）

1. **subscription-plans-card.tsx 中 API Plan 和 Coding Plan 的卡片渲染逻辑高度重复**

   - 文件：`web/default/src/features/wallet/components/subscription-plans-card.tsx:562-813`
   - 问题：两个 IIFE（API Plans 区域和 Coding Plan 区域）中的 Card 渲染结构几乎完全相同（标题、副标题、价格、Recommended badge、benefits 渲染、Subscribe 按钮），仅有 benefits 列表内容不同。约 200 行代码是纯粹的复制粘贴。
   - 为什么重要：后续修改卡片布局（如添加新的 UI 元素）需要在两处同步修改，维护成本高且容易遗漏。
   - 修复建议：提取一个通用的 `PlanCard` 组件，接收 `plan`、`isPopular`、`reached`、`benefits`、`onSubscribe` 等 props。两个区域只负责构建各自的 benefits 列表，然后调用 `PlanCard`。

2. **rate_limit_weekly_multiplier 的 UI 和 schema 约束不一致**

   - 文件：`web/default/src/features/subscriptions/components/subscriptions-mutate-drawer.tsx:612-615`（`min={1}`、`|| 1`），`web/default/src/features/subscriptions/lib/plan-form.ts:48`（`z.coerce.number().min(0).default(0)`）
   - 问题：Input 的 `min` 属性和 `onChange` 中的 `|| 1` 默认值阻止了用户输入 0，但 zod schema 允许 `min(0)`，即 0 是合法值。虽然从业务角度看 weekly_multiplier = 0 不太合理（等于禁用），但这种不一致可能在未来导致困惑。
   - 为什么重要：schema 是数据的真相来源，UI 应与 schema 对齐。如果业务上不应为 0，schema 也应 `min(1)`；如果业务上可以为 0，UI 不应阻止。
   - 修复建议：与业务方确认后统一。推荐 schema 改为 `min(1)`（因为 multiplier 为 0 无意义），或 UI 中 `|| 1` 改为 `|| 0`。

3. **fr/ja/ru/vi 语言文件中新增的 i18n key 未翻译**

   - 文件：`web/default/src/i18n/locales/fr.json`、`ja.json`、`ru.json`、`vi.json` 的 diff 部分
   - 问题：新增的 20 个 key 全部使用英文原文作为翻译值，未做本地化。例如 fr.json 中 `"Rate Limit": "Rate Limit"`、`"Common Fields": "Common Fields"`。
   - 为什么重要：这些 key 包含用户可见的 UI 文案（如 "Enter amount in M tokens"、"Weekly limit"），法语/日语/俄语/越南语用户将看到英文界面。
   - 修复建议：这可能是 i18n:sync 工具的自动行为（回退到英文原文），属于已知的技术限制。如果有翻译资源，应补充翻译。如果当前没有翻译资源，可在后续迭代中处理。

4. **plan-form.ts 中 total_amount 的 M token 转换不区分 plan_type**

   - 文件：`web/default/src/features/subscriptions/lib/plan-form.ts:88`（`total_amount: Number(plan.total_amount || 0) / 1_000_000`）和第 113 行（`total_amount: Math.round(Number(values.total_amount || 0) * 1_000_000)`）
   - 问题：`total_amount` 的转换对 api 和 coding_plan 都生效，但 coding_plan 的表单 UI 中不展示 total_amount 字段。当编辑 coding_plan 时，formValuesToPlanPayload 会将 planToFormValues 转换出的 M token 值再转回实际 token 数。虽然 roundtrip 结果正确，但如果 coding_plan 的 total_amount 在 UI 中不可见也不可编辑，将其做不必要的来回转换增加了出错风险。
   - 为什么重要：如果未来有人修改 total_amount 的转换逻辑，可能会忽略 coding_plan 场景下的隐性影响。
   - 修复建议：这不是 bug，当前 roundtrip 行为正确。建议在代码中添加注释说明 coding_plan 场景下 total_amount 也会被转换但不在 UI 中展示，保持数据一致性。
