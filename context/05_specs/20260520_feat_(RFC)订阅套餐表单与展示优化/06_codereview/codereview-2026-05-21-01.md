#### 重要（应该修复）

1. **5 个 i18n 翻译 key 缺失，运行时将显示英文 key 而非翻译文本**

   - 文件：`subscriptions-mutate-drawer.tsx` 第 245 行（`Common Fields`）、第 465 行（`Quota Settings`，已存在于 en.json 但 zh 翻译需确认 T5 补充）、第 481/584 行（`Enter amount in M tokens`）、第 487/590/633 行（`M token`）、第 631 行（`Weekly limit`）

   - 为什么重要：这不是 T2 代码本身的 bug，而是任务拆分导致的临时状态。`Quota Settings` 已存在于所有语言文件（zh: "额度设置"），所以 API Plan 分区标题正常。但 `Common Fields`、`M token`、`Enter amount in M tokens`、`Weekly limit` 在 en.json 中都不存在，会回退到 key 本身（恰好是英文），功能上不会崩溃，但中文环境下用户体验不佳。
   - 处理建议：这属于跨任务依赖问题，在 T5 完成后会自动解决。如果 T5 能紧接着 T2 执行，可以接受当前状态。如果 T5 可能延迟，建议将这 5 个 key 提前添加到 en.json/zh.json，或合并 T5 到 T2 的同一个 commit 中。

#### 次要（改进项）

1. **`rate_limit_weekly_multiplier` 的 UI 约束与 schema 定义不一致**

   - 文件：`subscriptions-mutate-drawer.tsx` 第 611-615 行
   - 问题：Input 设置了 `min={1}` 和 `parseInt(...) || 1`，暗示最小值为 1。但 `plan-form.ts` 第 48 行的 zod schema 是 `.min(0).default(0)`，默认值为 0。这意味着 schema 允许 0 值，但 UI 层强制至少为 1。
   - 影响：功能上不影响，因为 `rate_limit_weekly_multiplier` 只在 `coding_plan` 类型下显示和使用，且语义上倍数为 0 或 1 都合理。但 schema 和 UI 的约束不一致可能导致维护困惑。
   - 说明：这源自开发计划的设计，实现与计划一致，不是 T2 引入的新问题。

2. **`CalendarClock` 图标语义与 "Common Fields" 分区不完全匹配**

   - 文件：`subscriptions-mutate-drawer.tsx` 第 244 行
   - 问题：`CalendarClock` 图标原先用于 "Duration Settings" 分区，语义匹配。现在该分区扩展为 "Common Fields"（包含价格、有效期、升级组、排序等），图标仍沿用 `CalendarClock`，语义关联变弱。
   - 影响：纯视觉语义问题，不影响功能。更合适的图标可能是 `SlidersHorizontal` 或 `ListChecks`。
   - 说明：开发计划中指定了 `CalendarClock`，实现与计划一致。
